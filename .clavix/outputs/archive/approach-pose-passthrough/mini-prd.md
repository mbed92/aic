# Requirements: Approach Pose Passthrough Fix

*Generated from conversation on 2026-03-25*

## Objective

Fix a bug in `CompositeInsertCableStrategy` where the refined pose computed inside `ApproachStrategy.execute()` is silently discarded. The insertion stage always receives the original coarse pose from perception, ignoring any fine-tuning done during approach.

## Core Requirements

### Must Have (High Priority)

- **[HIGH]** Change `ApproachStrategy.execute()` return type from `bool` to `Pose | None` — `None` signals failure, a `Pose` value is the final position after approach (refined or original if no refinement occurred).
- **[HIGH]** Update `CompositeInsertCableStrategy.execute()` to capture the returned pose and pass it to `_insertion.execute()` instead of the original `port_pose`.
- **[HIGH]** Update `ROS2ServiceApproach.execute()` to return the final `port_pose` (after optional refinement).
- **[HIGH]** Update `ApproachStrategy` base class (`approach/base.py`) signature accordingly.

### Should Have (Medium Priority)

- **[MEDIUM]** The change eliminates the ambiguity between "approach failed" and "approach succeeded without refinement" — both previously returned `bool`, now expressed as `None` vs `Pose`.

### Could Have (Low Priority / Inferred)

- **[LOW]** Add a brief docstring note to `ApproachStrategy.execute()` explaining that the returned `Pose` is the authoritative position for the subsequent insertion stage.

## Technical Constraints

- **Framework/Stack:** Python, ROS2, existing strategy pattern architecture
- **Files affected:**
  - `strategies/approach/base.py` — abstract method signature
  - `strategies/approach/ros2_service.py` — concrete implementation
  - `strategies/composite.py` — caller
- **Backward compatibility:** No other callers of `ApproachStrategy.execute()` exist currently — safe to change signature.

## Architecture & Design

- **Pattern:** Strategy pattern — perception → approach → insertion pipeline
- **Key Decision:** Return type carries both status (`None` = failure) and data (`Pose` = final position), avoiding a separate out-parameter or mutable state.

## User Context

**Target Users:** Developers maintaining `romanelovect_policy`
**Primary Use Case:** Correct pose is passed from approach stage to insertion stage, preserving any fine-tuning done during close-range scanning.
**User Flow:** `_perception.estimate_port_pose()` → `_approach.execute()` returns final Pose → `_insertion.execute(final_pose, ...)`.

## Edge Cases & Considerations

- When `_refine()` returns `None` (not yet implemented / no refinement available), `ROS2ServiceApproach` must still return the original `port_pose`, not `None`.
- `None` return from `execute()` must unambiguously mean "approach failed", not "no pose available".

## Success Criteria

- ✓ `CompositeInsertCableStrategy` passes the pose returned by approach to insertion — not the original perception pose.
- ✓ `ROS2ServiceApproach.execute()` returns `refined_pose` when available, otherwise `port_pose`.
- ✓ All three files compile and existing smoke tests pass.
- ✓ When `_refine()` is later implemented, its output automatically feeds into insertion without further changes to composite or insertion code.
