# Requirements: Perception Strategy Split

*Generated from conversation on 2026-03-25*

## Objective

Rozbić istniejący `PerceptionStrategy` ABC na dwa niezależne interfejsy: `PointCloudStrategy` (generowanie pointclouda) i `BoardLocalizationStrategy` (lokalizacja portu w pointcloudzie). Motywacja: wymienność metody generowania pointclouda — system ma dostępne obrazy RGB i może używać SGBM lub modeli AI wymiennie, bez zmiany logiki lokalizacji.

## Core Requirements

### Must Have (High Priority)

- **[HIGH]** Stworzyć `PointCloudStrategy` ABC z metodą `generate(task, get_observation) -> PointCloud | None` — gdzie `PointCloud` to `open3d.geometry.PointCloud`.
- **[HIGH]** Stworzyć `BoardLocalizationStrategy` ABC z metodą `estimate_port_pose(task, point_cloud: PointCloud) -> Pose | None`.
- **[HIGH]** Stworzyć `CompositePerceptionStrategy` implementującą istniejący `PerceptionStrategy` ABC — przyjmuje `PointCloudStrategy` i `BoardLocalizationStrategy` przez konstruktor i orkiestruje ich wywołanie w `estimate_port_pose()`.
- **[HIGH]** `CompositeInsertCableStrategy` nie zmienia się — nadal przyjmuje `PerceptionStrategy`. Nie wie o dwuetapowości percepcji.
- **[HIGH]** Podzielić `SGBMICPPerception` (stub TODO) na dwa osobne stuby: `SGBMPointCloud` implementujący `PointCloudStrategy` i `ICPBoardLocalization` implementujący `BoardLocalizationStrategy`.

### Should Have (Medium Priority)

- **[MEDIUM]** `SGBMPointCloud` i `ICPBoardLocalization` powinny rzucać `NotImplementedError` (zgodnie z zasadą *fail loud not silent*), nie zwracać `None`.

### Could Have (Low Priority / Inferred)

- **[LOW]** Usunąć stary `PerceptionStrategy` ABC i `SGBMICPPerception` po zakończeniu migracji (lub zastąpić je nowymi).

## Technical Constraints

- **Framework/Stack:** Python 3.10+, ROS2, open3d (`open3d.geometry.PointCloud` jako format pośredni)
- **Files affected:**
  - `strategies/perception/base.py` — rozbudowa o nowe ABC
  - `strategies/perception/sgbm_icp.py` → podzielić na `strategies/perception/sgbm_point_cloud.py` + `strategies/perception/board_localization/icp.py` (lub podobna struktura)
  - `strategies/composite.py` — bez zmian (przyjmuje `PerceptionStrategy`)
  - Nowy plik: `strategies/perception/composite.py` — `CompositePerceptionStrategy`

## Architecture & Design

- **Pattern:** Strategy + Composite
- **Hierarchy:**
  ```
  PerceptionStrategy (ABC, istniejący)
      └── CompositePerceptionStrategy
              ├── PointCloudStrategy (ABC, nowy)
              │       └── SGBMPointCloud (stub)
              └── BoardLocalizationStrategy (ABC, nowy)
                      └── ICPBoardLocalization (stub)
  ```
- **Key Decision:** `CompositePerceptionStrategy` implementuje `PerceptionStrategy` — `CompositeInsertCableStrategy` nic nie wie o podziale na dwa etapy (single level of abstraction).
- **Key Decision:** Format pośredni to `open3d.geometry.PointCloud`, nie mapa głębokości ani surowe piksele.
- **Key Decision:** Oba nowe interfejsy są wymienialne niezależnie (DIP).

## Logic of `CompositePerceptionStrategy.estimate_port_pose()`

```
point_cloud = self._point_cloud.generate(task, get_observation)
if point_cloud is None:
    return None
return self._board_localization.estimate_port_pose(task, point_cloud)
```

## User Context

**Target Users:** Developerzy implementujący percepcję w `romanelovect_policy`
**Primary Use Case:** Podmiana SGBM na model AI (lub odwrotnie) bez zmiany logiki lokalizacji boarda ani pipeline'u insercji.
**User Flow:** `RomanelovectPolicy` komponuje `CompositePerceptionStrategy(SGBMPointCloud(), ICPBoardLocalization())` — zmiana na AI to podmiana jednego argumentu konstruktora.
